## intro

gorkie is a helpful ai slack assistant.
it started as "answer questions, help with tasks, and be useful in threads," then users started asking for stuff that is not just text.

like:

* "run this script"
* "convert this file"
* "download this thing"
* "inspect this repo"
* "do the annoying terminal thing for me"

at some point, you either say "sorry, i can't" or you give it a sandbox and let it cook. so, sandboxes.

## what stayed constant

from phase 1 onward, a few things were consistently important regardless of provider/agent:

* syncing attachments into the sandbox
* making tool status visible (so you can tell what it's doing, not just "thinking…" forever)
* streaming updates back to slack instead of waiting for one giant final blob

we use **slack's ai implementation** for the chat surface and interaction model, which lets us render **interactive ui** directly in slack:
* https://docs.slack.dev/ai/
* https://docs.slack.dev/reference/block-kit/blocks/plan-block/

## phase 1: vercel sandboxes

we were already using vercel's [ai sdk](https://ai-sdk.dev/). [vercel sandboxes](https://vercel.com/docs/vercel-sandbox/) are isolated execution environments you can spin up on demand, and they had a generous free tier, so we tried the obvious combo.

we gave gorkie a simple sandbox tool that spawned a **ToolLoopAgent** with tools to work in the sandbox.

the tool loop agent felt nice conceptually:

1. model decides it needs to execute something
2. it calls a "sandbox" tool
3. the sandbox runs a subagent and completes the task.
4. results go back to gorkie, then to slack

it worked, but it was also kinda janky.

### what sucked

vercel sandboxes felt "fine for quick runs," but not like "serious execution infra."

things we missed or kept bumping into:

* not as advanced as e2b/daytona-style sandboxes
* lifecycle controls felt limited
* snapshot support was not where we wanted it
* random bugs and weirdness
* limits were easy to hit (bandwidth, storage, etc.)

it was enough to prove that "yes, sandboxes make gorkie way more useful," but it was not the foundation.

## phase 2: e2b

next we moved to **e2b** ([e2b.dev](https://e2b.dev/)).
**e2b** is a sandbox platform built for ai agents, and it felt more "real" as a provider for longer-running execution.

we kept the same overall architecture (tool loop agent), just swapped the sandbox layer.

honestly, it went pretty well.

users said it worked fine.

### but then reality hit

as usage grew, it became clear the provider itself was not the biggest issue. the bigger issue was our **tool loop architecture**.

it could handle short tasks, but it struggled with long-term runtime needs:

* compaction
* skills
* mcp plumbing
* coordination and orchestration
* stateful session handling
* retries and failure recovery without spaghetti

in other words, we were gradually building our own runtime by accident.

we also had provider-level friction with **e2b**:

* we had to keep extending timeouts
* persistence existed, but it was beta and sometimes buggy
* the model felt closer to "create sandbox, kill sandbox" than true persistence

what we wanted was a persistence-first setup from day one.

## state + persistence

we persist sandbox state in our database so we can reattach per thread. the core idea is:

1. `threadId → sandboxId + sessionId`
2. status transitions (`active`, `paused`, `resuming`, `error`, `destroyed`)
3. reattach on the next message in the same thread

## phase 3: daytona

so we switched to **[daytona](https://www.daytona.io/)**. **daytona** is a sandbox/runtime platform with strong lifecycle controls, which matched our persistence-first model better.

daytona was the right fit because:

* great dx (developer experience)
* built around persistence
* lifecycle felt cleaner
* it matched the model we wanted: “this thread owns a runtime”

we set it up so each thread gets its own sandbox, and it persists while the thread stays active. we also store `threadId → sandboxId + sessionId`, so a thread always reattaches to the same runtime.

daytona's lifecycle controls made cleanup easy:

* auto-stop after **5 minutes** of inactivity
* auto-archive **2 hours** after stop
* auto-delete after **2 days**

no custom janitor. just set the parameters and let the platform clean up.

once we picked daytona, the next question was: what coding agent do we run inside the sandbox? we definitely did not want to keep building all of that orchestration ourselves.

## phase 4: sandbox agent + opencode

we stumbled onto [sandbox agent](https://sandboxagent.dev/), a project by [rivet.dev](https://rivet.dev).
**sandbox agent** runs inside the sandbox and exposes an agent interface we can talk to. the idea was:

* sandbox agent runs inside the sandbox
* it runs a coding agent inside it
* it exposes an http server in the sandbox
* we talk to that server using the sandbox agent sdk
* communication uses [acp](https://zed.dev/acp) (an agent communication protocol)

we first tried **[opencode](https://opencode.ai/)** behind it.

it worked like:

### why it was better than tool loop agent

because it offloaded a bunch of orchestration stuff to something purpose-built.

### why we still didn't love it

it introduced a whole new set of failure points:

* an http server inside the sandbox
* more moving parts
* more "if this dies, everything dies"
* more complexity than we wanted

also, opencode felt a bit over-engineered for our needs.
it used more tokens than we wanted.
we wanted simple and efficient.

## phase 5: pi with sandbox agent

so we tried **[pi](https://pi.dev/)**, a coding agent, with **sandbox agent**.

pi is lightweight and extensible, and it supports the things we cared about:

* skills
* mcps
* being easy to extend

it worked decently, but we still had the same core issue: we were stuck with the "http server inside sandbox" shape, which we already felt was overcomplicating things. we also still hit reliability issues sometimes. crashes happen, life happens, but we wanted boring infra.

## phase 6: pi rpc

then we found the cleanest path.

at this point, we had two integration paths: acp/http server mode or **rpc**.
we picked **rpc** with pi.

because rpc is just:

* send json
* get json back

no server, no extra daemon, no "did the http listener die?" problems, just a process.

### what we built

we made a small rpc client inspired by pi's own client.

**the flow**:

1. gorkie decides it needs execution
2. we spin up or attach to the thread's daytona sandbox
3. we start pi in rpc mode
4. our rpc client sends:
   * prompt
   * context
5. pi runs commands, edits files, and does agent stuff
6. rpc returns structured output
7. we stream it back into slack using slack assistant api

**what the rpc path actually looks like**:

* we launch `pi --mode rpc` inside a **PTY**
* PTY echo is disabled so JSON doesn't get mirrored back
* messages are **newline-delimited JSON**
* resume uses `pi --mode rpc --session <id>`

it was simple, fast, and reliable. it didn't feel like a house of cards anymore.

## tools with pi

one of the biggest wins with pi was how easy tool integration became.

before, with sandbox agent + opencode, we had to do this weird dance:

* define a custom mcp server
* have the mcp server call our tool
* intercept the tool call
* route it
* pray

with pi, we can just:

* define a custom tool
* plug it in
* add status
* add metadata
* extend it without inventing new plumbing

## lessons learned

the stuff we wish we knew earlier:

* sandboxes are easy, orchestration is the real boss fight
* "just build it" turns into "we built a runtime" way too fast
* persistence-first is a different world than spin-up/spin-down
* fewer moving parts wins every time
* running an http server inside your sandbox is cool until it isn't

## closing

sandboxes basically upgraded gorkie from a "helpful chatbot" to something that "actually can do stuff."

moving to daytona + pi rpc made it:

* simpler
* cheaper (tokens and complexity)
* more reliable
* easier to extend

now we can focus on making features, not babysitting infra.

vercel launched a new **Chat SDK** that supports multiple runtimes. we might rewrite that layer in the future as things evolve.
